How Kubernetes Orchestrates the Cloud

Kubernetes (K8s) is an open-source platform that automates container operations—eliminating manual processes involved in deploying, scaling, and managing containerized applications.

The Anatomy

The Cluster Architecture

A Kubernetes cluster consists of two main parts: The Control Plane (the brain) and the Worker Nodes (the muscle).

Control Plane (Brain)

API Server

The entry point and gatekeeper of the cluster. All administrative communication passes through it.

etcd

A highly available and secure key-value store. It holds the complete state of your cluster at any given point.

Scheduler

Matches newly created Pods that don't have a node assigned yet to the best fit worker node in the cluster.

Controller Manager

Runs continuous loop controllers that regulate the state of the cluster, checking if the actual state matches the desired state.

Worker Nodes (Muscle)

Kubelet

The primary node agent. It reports node status back to the API Server and makes sure that containers in Pods are running.

Container Runtime

The software responsible for actually running the containerized applications (usually containerd or Docker).

Kube-Proxy

An essential network proxy running on each node, maintaining network rules to allow communication to your Pods from inside or outside the cluster.

Glossary

Core Kubernetes Concepts

Learn the fundamental building blocks of Kubernetes resources and objects.

  • Pods
  • Deployments
  • Services
  • Ingress
  • Namespaces

What is a Pod?

Pods are the smallest, most basic deployable objects in Kubernetes. A Pod represents a single instance of a running process in your cluster and can contain one or more containers.

  • Shared Resources: All containers in a Pod share the same network IP, port space, and storage volumes.
  • Ephemeral Nature: Pods are created, run, and destroyed. They are not self-healing on their own unless managed by a controller.
  • One-to-One Model: The most common pattern is one container per Pod, but sidecar helpers are often deployed alongside the main app.

What is a Deployment?

A Deployment provides declarative updates for Pods and ReplicaSets. You describe a desired state in a Deployment, and the Deployment Controller changes the actual state to the desired state at a controlled rate.

  • Replica Management: Manages how many duplicate copies (replicas) of a Pod should run across nodes.
  • Rolling Updates: Upgrades application code without downtime by progressively starting new pods while shutting down old ones.
  • Rollbacks: Allows rolling back to a previous Deployment revision if the current state is unstable.

What is a Service?

An abstract way to expose an application running on a set of Pods as a network service. Since pods are volatile, their IPs change constantly. Services act as stable endpoints.

  • Static IP & DNS: Provides a single, unchangeable IP address and DNS name for a set of target Pods.
  • Load Balancing: Distributes network traffic uniformly across all available target pods matching the Service selector.
  • Selector matching: Dynamically routes traffic using labels (e.g. `app: web`), tracking pod additions and removals automatically.

What is Ingress?

An API object that manages external access to the services in a cluster, typically HTTP. Ingress can provide load balancing, SSL termination, and name-based virtual hosting.

  • Routing Rules: Allows exposing multiple internal cluster services under a single external IP address via HTTP paths (e.g. `/api` or `/web`).
  • Domain Management: Map domains like `example.com` directly to internal Services.
  • SSL/TLS termination: Decrypt secure HTTPS traffic at the perimeter before routing it internally.

What is a Namespace?

Namespaces are virtual clusters backed by the same physical cluster. They provide a scope for names and allow dividing cluster resources between multiple users, projects, or environments.

  • Resource Isolation: Ideal for separating dev, staging, and production environments within a single cluster.
  • Access Control: Secure individual workspaces using RBAC (Role-Based Access Control) defined on specific namespaces.
  • Quota Management: Limit CPU, memory, and storage volumes allocated to specific teams or environments.
Capabilities

Why Teams Choose Kubernetes

Kubernetes brings resilient infrastructure, automation, and standardization to cloud-native applications.

Self-Healing

K8s automatically restarts containers that fail, replaces and reschedules containers when nodes die, and kills containers that don't respond to user-defined health checks.

Horizontal Scaling

Scale your application up or down automatically based on CPU usage, memory usage, or custom metrics, via commands or directly through the UI.

Load Balancing

K8s exposes containers using DNS names or static IPs. If traffic to a container is high, it distributes the network load evenly, ensuring stable performance.

Automated Rollouts

Declare the desired state for your containerized apps, and K8s rolls them out at a controlled rate, automatically rolling back changes if errors occur.
